Fastest · ~90 sec
If the 1-click deploy fails with “Authentication error [code: 10000]” — fix in 60 sec

Cloudflare’s auto-generated API token for new Pages projects sometimes lacks the Pages:Edit scope, so the first build can’t deploy. You’ll see the auth error in the build log. Fix:

  1. Create a token at dash.cloudflare.com/profile/api-tokens with these Account permissions: Cloudflare Pages:Edit, D1:Edit, Workers R2:Edit, Workers AI:Edit, Workers Scripts:Edit, Account Settings:Read.
  2. In the dashboard: Workers & Pages → your project → Settings → Build & deployments → API token → paste it.
  3. Re-trigger the deploy from the Deployments tab.

The CLI installer below avoids this entirely — it uses wrangler login (OAuth) rather than an API token.

Hit a snag.

Cloudflare can't see your repo. Even with your GitHub linked to Cloudflare, the linkage is per-repo — Cloudflare can only pull from repos owned by you that have Cloudflare Workers & Pages granted access. Three things to verify, in order:

  1. Does the repo exist? We're looking for —. Open it → — if you get a 404, you haven't forked yet. Fork now
  2. Did you grant the Cloudflare app access to it? Manage permissions → — choose your fork (or "All repositories") and Save.
  3. Are you using the right Cloudflare account? The API token has to belong to the same Cloudflare account where you authorised the GitHub App. If you have multiple, double-check which one your token is from.

Once all three are true, click Resume install below. Your D1 + R2 from the earlier steps are kept; only the Pages project is recreated.
